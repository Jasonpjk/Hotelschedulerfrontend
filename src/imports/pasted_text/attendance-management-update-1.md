현재 LOTTE HOTELS & RESORTS 프런트 스케줄 SaaS의 “근태 관리” 화면에서
근태 신청 처리 방식과 근무표 반영 흐름을 정리해줘.

이번 작업의 핵심은 아래 3가지야.

1) 근태 관리 목록 컬럼 정리
2) SL을 연차로 변경할 때 하단 잔여 수량 즉시 연동
3) 교육/병가 신청 승인 시 운영정책을 반영해 자동 재배치하고 새 근무표 버전으로 저장

이번 작업은 근태 관리와 근무표 반영 흐름을 보완하는 작업이야.
기존 근무표, 직원 관리, 대시보드, 연간 야간조 계획, 알림, 변경 이력 기능은 삭제하거나 롤백하지 마.

--------------------------------------------------
[수정 1. 근태 관리 목록 컬럼 정리]
--------------------------------------------------

현재 근태 관리 직원별 목록에 “신청 날짜 요약” 컬럼이 있는데
이 컬럼은 삭제해줘.

신청 날짜는 상세 검토 모달 안에서 확인하면 되므로
직원별 목록에서는 필요 없음.

직원별 목록 컬럼 순서는 아래처럼 정리해줘.

권장 컬럼 순서:
- 직원명
- 성별
- 신청 건수
- 자동 반영
- 미반영
- 검토 필요
- 휴가 잔여
- 연차 잔여
- HO 사용
- 관리

중요:
- “신청 날짜 요약” 컬럼은 제거
- 컬럼 순서는 “신청 건수 → 자동 반영 → 미반영 → 검토 필요” 순서로 정리
- 기존 표 디자인 톤앤매너는 유지
- 표 헤더와 숫자 정렬이 깨지지 않게 조정

--------------------------------------------------
[상태 용어 기준]
--------------------------------------------------

근태 관리의 상태 기준을 아래처럼 명확히 구분해줘.

1) 자동 반영
- 시스템 정책에 따라 근무표에 이미 반영된 건
- 예: SL, 연차, 휴가, EDU, SICK 등으로 반영 완료

2) 미반영
- 신청은 접수되었지만 근무표에 아직 아무 코드도 반영되지 않은 건
- 예: 최소 인원 부족, 충돌, 승인 전 상태 등으로 실제 근무표 미반영

3) 검토 필요
- 관리자가 확인하거나 판단해야 하는 모든 건
- 미반영뿐 아니라 관리자 조정 필요, 승인 대기, 충돌, SL → 연차 변경 가능 건 포함

중요:
- 검토 필요 = 미반영이 아님
- 미반영은 검토 필요의 일부일 수 있음
- SL로 이미 반영되었지만 연차로 바꿀 수 있는 건은 “미반영”이 아니라 “검토 필요” 또는 “관리자 조정 필요”로 봐야 함

--------------------------------------------------
[수정 2. SL → 연차 변경 시 잔여 수량 즉시 연동]
--------------------------------------------------

상세 검토 모달에서 SL로 자동 반영된 건을 관리자가 “연차로 변경”할 수 있음.

이때 단순히 표기만 바꾸지 말고 아래 값이 모두 즉시 연동되어야 함.

변경 전 예시:
- 최종 반영: SL
- 이번 달 SL 사용: 사용함
- 연차 잔여: 15 / 15일
- 상태: 관리자 조정 필요

관리자가 “연차로 변경” 클릭 후:
- 최종 반영: 연차 또는 VAC
- 이번 달 SL 사용: 사용 안 함 또는 미사용
- 연차 잔여: 14 / 15일
- 근무표 셀 코드: SL → VAC 또는 연차 코드
- 상태: 자동 반영 완료 또는 관리자 조정 완료
- 변경 이력 기록

중요:
- SL → 연차 변경 시 하단의 “이번 달 SL 사용”, “연차 잔여” 숫자가 즉시 재계산되어야 함
- 화면을 새로고침해야만 바뀌면 안 됨
- 해당 직원의 근태 관리 목록 숫자도 함께 갱신
- 대시보드의 검토 필요 / 미반영 / 자동 반영 숫자도 같은 기준으로 갱신

변경 이력 예시:
- 김민준 / 04-05 / SL → VAC
- 사유: 관리자 판단에 따라 SL을 연차로 변경
- 수정자: 김재민 Administrator

--------------------------------------------------
[연차 코드 표기 기준]
--------------------------------------------------

현재 근무표 범례에는 VAC 휴가 코드가 있음.
화면 문구에서는 “연차”라고 표현하되,
근무표 셀 코드로는 기존 VAC 체계를 사용할 수 있음.

권장:
- 근무표 코드: VAC
- 사용자 표시 문구: 연차
- 버튼 문구: 연차로 변경

단, 기존 코드 체계를 깨지 말고 현재 시스템에서 사용하는 연차/휴가 코드 기준과 맞춰줘.

--------------------------------------------------
[수정 3. 교육 신청 승인 기능 추가]
--------------------------------------------------

현재 교육 신청 건은 승인 대기까지만 있고
실제 승인 버튼과 근무표 반영 기능이 없음.

교육 신청 건에는 아래 액션을 추가해줘.

주요 버튼:
- 승인 및 자동 재배치
- 반려

기본 흐름:
1) 직원이 교육 신청
2) 근태 관리 상세 검토에서 관리자가 확인
3) 관리자가 “승인 및 자동 재배치” 클릭
4) 시스템이 EDU 반영 시뮬레이션 실행
5) 운영정책 기준으로 자동 재배치 가능 여부 판단
6) 가능하면 새 근무표 버전 저장
7) 교육 신청 상태를 승인 완료로 변경
8) 불가능하면 반려 또는 관리자 수동 조정 필요 상태로 안내

--------------------------------------------------
[교육 승인 시뮬레이션]
--------------------------------------------------

교육 신청을 승인할 때는 바로 현재 근무표를 덮어쓰지 말고
먼저 시뮬레이션을 실행해줘.

교육 승인 시뮬레이션에서 확인할 항목:
- 해당 날짜에 EDU를 넣을 수 있는지
- 기존 근무 코드가 있다면 EDU로 교체 가능한지
- 해당 직원이 빠져도 최소 인원이 유지되는지
- 각 조별 인차지 조건이 유지되는지
- 수요 예측 기준 인원 부족이 발생하지 않는지
- 대체 가능한 직원이 있는지
- 대체 직원의 휴가/병가/교육/SL/HO와 충돌하지 않는지
- A13 다음날 M07 같은 고강도 패턴이 생기지 않는지
- 연속근무 기준을 위반하지 않는지
- 전월 근무강도 회복 대상자에게 무리한 배치가 생기지 않는지

--------------------------------------------------
[교육 승인 가능 시 동작]
--------------------------------------------------

시뮬레이션 결과 교육 반영이 가능하면
결과 확인 모달을 보여줘.

예시:
- 교육 반영 가능
- 신청자: 오세영
- 대상일: 04-28
- 기존 근무: M07
- 변경 후: EDU
- 대체 배치: 박지호 OFF → M07
- 최소 오전조 인원 충족
- 인차지 조건 충족
- 추가 경고 없음

모달 버튼:
- 새 버전으로 저장하고 승인 완료
- 취소

관리자가 “새 버전으로 저장하고 승인 완료” 클릭 시:
- 현재 버전을 직접 덮어쓰지 말고 새 근무표 버전 생성
- 신청자의 해당 날짜 셀을 EDU로 반영
- 대체 직원의 근무표도 재배치 결과 반영
- 교육 신청 상태: 승인 완료
- 근무표 반영 상태: 관리자 반영 완료 또는 자동 재배치 완료
- 변경 이력 기록
- 공통 toast 표시

새 버전명 예시:
- v4.1 교육 반영 자동재배치
또는 기존 버전 규칙에 맞춰 자동 생성

버전 설명 예시:
- 생성 사유: 오세영 04-28 교육 승인 및 자동 재배치
- 변경 내용:
  - 오세영 04-28 M07 → EDU
  - 박지호 04-28 OFF → M07
  - 최소 오전조 인원 충족
  - 인차지 조건 유지

--------------------------------------------------
[교육 승인 불가 시 동작]
--------------------------------------------------

시뮬레이션 결과 도저히 교육 반영이 불가능하면
바로 승인하지 말고 불가 사유를 보여줘.

교육 반영 불가 조건 예:
- 최소 인원 충족 불가
- 인차지 배치 불가
- 대체 가능한 직원 없음
- 휴가/병가/교육/SL/HO와 충돌
- 해당일 수요예측 피크로 인원 부족
- 재배치 시 연속근무 또는 고강도 패턴 과다 발생

불가 모달 예시:
- 교육 반영 불가
- 해당일 오전조 최소 인원 및 인차지 조건을 충족할 수 없습니다.
- 대체 가능한 직원이 없습니다.
- 교육 일정을 조정하거나 신청을 반려해야 합니다.

모달 버튼:
- 반려 처리
- 관리자 수동 조정
- 취소

반려 처리 시:
- 신청 상태: 반려
- 반려 사유 자동 입력 가능
- 직원 화면에도 반려 사유 표시
- 근무표에는 EDU 반영하지 않음

반려 사유 예시:
- 해당일 최소 운영 인원 및 인차지 배치 기준을 충족할 수 없어 교육 신청을 반려합니다.

--------------------------------------------------
[수정 4. 병가 신청 승인 및 긴급 재배치]
--------------------------------------------------

병가는 교육과 다르게 처리해야 함.
병가는 갑작스러운 건강 사유이므로 운영상 어렵더라도 기본적으로 승인 후 긴급 재배치를 시도하는 구조가 적절함.

병가 신청 건에는 아래 액션을 추가해줘.

주요 버튼:
- 승인 및 긴급 재배치
- 반려

단, 반려는 증빙 미비 또는 정책상 병가 요건 불충족 같은 경우에만 사용하도록 설계.

--------------------------------------------------
[병가 승인 흐름]
--------------------------------------------------

병가 승인 기본 흐름:
1) 직원이 병가 신청
2) 근태 관리 상세 검토에서 관리자가 확인
3) 관리자가 “승인 및 긴급 재배치” 클릭
4) 해당 날짜 SICK 반영 시뮬레이션 실행
5) 운영정책 기준으로 긴급 재배치 시도
6) 가능하면 새 근무표 버전 저장
7) 일부 부족이 남으면 경고 포함 새 버전 저장 + 관리자 추가 조정 필요
8) 병가 신청 상태는 승인 완료로 변경

--------------------------------------------------
[병가 승인 가능 시 동작]
--------------------------------------------------

병가 반영이 가능하면:
- 신청자의 해당 날짜 근무표 셀을 SICK로 변경
- 기존 근무 코드가 있다면 SICK로 교체
- 빠진 인원을 운영정책에 따라 자동 대체
- 새 근무표 버전 생성
- 병가 신청 상태: 승인 완료
- 근무표 반영 상태: 긴급 재배치 완료
- 변경 이력 기록
- 공통 toast 표시

예시:
- 김민준 04-12 A13 → SICK
- 이서연 04-12 OFF → A13
- 최소 오후조 인원 충족
- 변경 이력 기록
- v4.2 병가 반영 긴급재배치 생성

--------------------------------------------------
[병가 승인 후 일부 경고가 남는 경우]
--------------------------------------------------

병가는 교육과 달리 재배치가 완벽하지 않더라도
승인 자체는 유지해야 하는 경우가 많음.

따라서 자동 재배치를 시도했지만 일부 문제가 남으면:
- SICK는 반영
- 가능한 범위에서 자동 재배치
- 새 버전 저장
- 운영 경고에 즉시 수정 필요 항목 추가
- 근태 관리 상태: 승인 완료 + 관리자 추가 조정 필요

예시:
- 병가 승인 완료
- SICK 반영 완료
- 단, 오전조 최소 인원 1명 부족
- 운영 경고에 즉시 수정 필요 추가
- 관리자 수동 조정 필요

--------------------------------------------------
[병가 반려 기준]
--------------------------------------------------

병가 반려는 운영 인원이 부족하다는 이유만으로 자동 처리하지 마.
병가는 교육과 다르게 건강 사유이므로
운영 인원 부족은 반려 사유가 아니라 긴급 조정 사유임.

병가 반려 가능 예:
- 증빙 필수 정책인데 증빙 미제출
- 병가 요건 불충족
- 잘못된 신청
- 관리자 판단에 따른 반려

반려 시:
- 근무표에 SICK 반영하지 않음
- 직원에게 반려 사유 표시

--------------------------------------------------
[자동 재배치 범위]
--------------------------------------------------

교육/병가 승인 후 자동 재배치는 전체 월을 무조건 다시 짜지 말고
영향 범위를 단계적으로 넓혀줘.

재배치 범위 우선순위:
1) 해당 날짜만 대체 인력 탐색
2) 해당 주간 범위에서 교환 탐색
3) 7일~14일 범위에서 운영정책 기반 재배치
4) 그래도 불가능하면 관리자 수동 조정 필요
5) 월 전체 재생성은 최후의 수단 또는 별도 확인 후 진행

중요:
- 교육/병가 1건 때문에 월 전체 근무표가 크게 바뀌면 안 됨
- 변경 범위는 최소화
- 변경된 셀 목록을 결과 모달에서 보여줘

--------------------------------------------------
[운영정책 반영 기준]
--------------------------------------------------

자동 재배치 시 호텔 설정의 운영정책을 반드시 반영해줘.

반영할 정책:
- 최소 조별 인원
- 각 조별 인차지 1명 이상
- A13 다음날 M07 지양
- 5일/6일 이상 연속근무 경고
- 전월 근무강도 회복 반영
- 야간조 전환 OFF 정책
- 교육 신청 승인 정책
- 병가 증빙 필수 여부
- 병가 반일 허용 여부
- 공휴일에도 인원 배치 필요

[휴가/연차/HO/SL 잔여와의 관계]

- HO 사용 차감 없음
- SL 사용 차감 없음

단, SL → 연차 변경은 연차 잔여에 영향을 줘야 함.
