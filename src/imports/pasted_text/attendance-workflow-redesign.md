현재 LOTTE HOTELS & RESORTS 관리자용 SaaS의 기존 톤앤매너를 유지한 상태로
“근태 신청”과 “근태 관리”를 서로 연결된 하나의 운영 흐름으로 재설계해주세요.

중요:
- 기존 시스템의 프리미엄 호텔 운영 관리자용 SaaS 톤앤매너 유지
- 딥 네이비 / 웜 화이트 / 절제된 골드 포인트 유지
- 넓은 여백, 얇은 선, 정돈된 그리드, 실무형 UI 유지
- 소비자용 일정 앱이 아니라 호텔 운영 시스템처럼 보여야 함

[1. 왼쪽 메뉴 구조 수정]
왼쪽 메뉴에서 “근태 관리” 위에 “근태 신청” 메뉴를 추가해주세요.

메뉴 순서 예시:
- 대시보드
- 근무표
- 직원 관리
- 근태 신청
- 근태 관리
- 수요 예측
- 호텔 설정

권한별 메뉴 활성화:
1) 일반 직원 로그인 시
- “근태 신청”만 활성화
- 나머지 관리자 메뉴는 비활성 또는 숨김 처리

2) 관리자 로그인 시
- “근태 관리” 사용 가능
- 필요 시 근태 신청 현황도 함께 확인 가능

즉,
직원은 신청만 하고
관리자는 전체 결과와 예외를 관리하는 구조로 설계해주세요.

[2. 전체 운영 원칙]
근태 신청과 근태 관리는 아래 우선순위를 기준으로 연결되어야 합니다.

우선순위:
1. 14일 동안 4일 휴무 규칙 충족
2. 수요 예측 기반 최소 운영 인원 충족
3. 공휴일 운영 필요 인원 판단
4. 직원 신청(휴가 / SL / 연차) 반영

중요:
직원 신청이 가장 먼저 반영되는 구조가 아니라,
호텔 운영과 최소 인원 기준이 먼저 충족된 후
가능한 범위에서 직원 신청이 반영되는 구조여야 합니다.

[3. 공휴일 처리 원칙]
공휴일은 무조건 쉬는 날이 아닙니다.

반드시 아래 원칙으로 반영해주세요.
- 공휴일이라도 호텔 운영이 우선
- 수요 예측 결과와 최소 운영 인원 기준을 먼저 판단
- 최소 인원만 출근해도 되는 상황이면 일부 직원은 공휴로 쉬도록 자동 반영
- 최소 인원 충족이 어려우면 공휴일에도 근무하도록 배치 가능

즉,
공휴일은 직원 신청이나 절대 우선 조건이 아니라
운영 상황에 따라 시스템이 자동 판단하는 조건으로 설계해주세요.

중요:
- 공휴일은 직원이 신청하는 항목이 아님
- 공휴일은 자동 스케줄 엔진이 배정하는 값
- “공휴를 먼저 보장하고 나머지를 채우는 방식”이 아니라
  “운영 최소인원 판단 후 가능할 때 공휴 반영” 구조로 가야 함

[4. 근태 신청 화면 구조]
“근태 신청”은 일반 직원이 본인 계정으로 로그인해서 사용하는 화면입니다.

화면 목적:
- 본인이 쉬고 싶은 날을 달력에서 선택해 신청
- 신청 가능한 유형은 아래 3개만 허용
  - 휴가
  - SL
  - 연차

중요:
- 공휴 / 휴무는 직원이 직접 신청하지 않음
- 공휴 / 휴무는 자동 스케줄 엔진이 처리

근태 신청 화면 핵심 구성:
- 상단: 페이지명, 연/월 선택
- 내 잔여 휴가 표시
- 내 잔여 연차 표시
- 달력형 신청 UI
- 날짜 선택
- 유형 선택(휴가 / SL / 연차)
- 신청 버튼
- 하단: 내 신청 내역 리스트

신청 폼 항목:
- 날짜
- 유형(휴가 / SL / 연차)
- 사유

중요:
- 사유는 필수가 아니라 선택 입력으로 처리
- 사유를 안 적어도 신청 가능해야 함
- 따라서 UI에서 “사유(선택)”처럼 명확히 보여주세요

[5. 근태 신청 상태 구조]
직원이 신청한 건은 단순 승인/반려보다 아래처럼 자동 반영 결과 중심으로 보이게 해주세요.

권장 상태:
- 신청 접수
- 자동 반영 가능
- 자동 반영 완료
- 검토 필요
- 반영 불가

즉,
직원 입장에서는 본인 신청이
- 접수되었는지
- 자동 반영되었는지
- 운영 조건상 반영이 어려운지
알 수 있어야 합니다.

[6. 자동 스케줄 반영 흐름]
근태 신청과 근태 관리는 아래 순서로 연결되어야 합니다.

자동 처리 흐름:
1. 수요 예측 데이터 반영
2. 공휴일 데이터 반영
3. 직원 마스터 / 역할 / 배치 그룹 반영
4. 시스템이 먼저 14일 동안 4일 휴무 규칙을 기준으로 기본 스케줄 생성
5. 최소 운영 인원 충족 여부 판단
6. 공휴일도 운영 가능 여부에 따라 자동 배치
7. 그 이후 직원이 신청한 휴가 / SL / 연차를 가능한 범위 내 자동 반영
8. 충돌 건은 별도 예외 처리 대상으로 분류

즉,
기본 스케줄 생성이 선행되고
그 위에 직원 신청을 덧씌우는 구조로 설계해주세요.

[7. 근태 관리 화면 재정의]
“근태 관리”는 관리자 승인 버튼을 일일이 누르는 화면보다는
자동 반영 결과와 예외 상황을 관리하는 화면으로 재구성해주세요.

즉,
기존의 단순 승인/반려 중심 화면이 아니라
“자동 반영 결과 + 예외 관리 워크스페이스”처럼 보여야 합니다.

근태 관리 화면에서 보여야 할 핵심 정보:
- 직원명
- 날짜
- 신청 유형
- 사유(있으면 표시, 없으면 비워도 됨)
- 자동 반영 결과
- 충돌 여부
- 운영상 검토 필요 여부
- 휴가 잔여
- 연차 잔여
- 최종 상태
- 관리 액션

중요:
근태 관리 담당자는 직원별로 아래 정보를 확인할 수 있어야 합니다.
- 남은 휴가 수량
- 남은 연차 수량

즉,
근태 관리 화면에서는 신청 건만 보는 것이 아니라
직원별 잔여 휴가/연차도 함께 볼 수 있어야 합니다.

[8. 근태 관리 화면의 핵심 역할]
관리자는 모든 신청을 수동 승인하는 사람이 아니라
자동 반영이 어려운 예외 상황을 조정하는 사람처럼 보여야 합니다.

따라서 근태 관리 화면은 아래 구조를 권장합니다.
탭 예시:
- 전체 신청
- 자동 반영 완료
- 검토 필요
- 반영 불가

또는 필터 예시:
- 전체 상태
- 자동 반영 완료
- 검토 필요
- 반영 불가

중요:
- 충돌 없는 신청 건은 자동 반영 가능
- 운영 커버리지 부족 등 예외 상황만 관리자 검토 대상
- 관리자 업무량을 줄이는 방향으로 설계

[9. 잔여 수량 반영 기준]
휴가와 연차는 승인 버튼을 누를 때 차감되는 구조보다
자동 반영 완료 또는 근무표 확정 시점에 반영되는 구조로 정리해주세요.

화면상 표현 방향:
- 직원 신청 화면: 내 잔여 휴가 / 내 잔여 연차 표시
- 근태 관리 화면: 직원별 휴가 잔여 / 연차 잔여 표시

중요:
- SL은 잔여 개수 중심보다 신청/상태 관리 중심
- 휴가 / 연차만 잔여 수량이 분명하게 드러나게

[10. 일반 직원용 근태 신청 UX]
일반 직원이 사용하는 화면은 복잡하면 안 됩니다.

직원용 UX 방향:
- 달력에서 쉬고 싶은 날짜 클릭
- 휴가 / SL / 연차 중 선택
- 사유는 선택 입력
- 신청 제출
- 내 신청 결과 확인

직원 화면에서 보이면 좋은 정보:
- 내 잔여 휴가
- 내 잔여 연차
- 이미 신청한 날짜
- 현재 상태
- 자동 반영 결과

[11. 관리자용 근태 관리 UX]
관리자용 화면은 운영 판단 중심이어야 합니다.

관리자 화면에서 보이면 좋은 정보:
- 어떤 신청이 자동 반영되었는지
- 어떤 신청이 운영상 충돌하는지
- 어떤 직원의 휴가/연차 잔여가 얼마인지
- 어떤 날이 수요 예측상 피크인지
- 어떤 공휴일에 최소 인원 충족이 어려운지

즉,
근태 관리 화면은 단순 목록이 아니라
운영과 연결된 판단형 화면처럼 보여야 합니다.

[12. 디자인 방향]
반드시 아래 조건을 지켜주세요.
- 기존 대시보드 / 근무표 / 직원 관리 / 수요 예측과 동일한 디자인 시스템처럼 보이게
- 관리자 화면은 차분하고 정제되게
- 일반 직원용 근태 신청 화면도 같은 브랜드 시스템 안에 있는 화면처럼 보이게
- 과한 앱 느낌, 캐주얼한 캘린더 앱 느낌 금지
- 실무형 상태 태그와 결과 중심 UI
- 고급스럽고 차분한 컬러 사용

[13. 최종 요청]
현재 시스템을 아래 방향으로 재설계해주세요.

1. 왼쪽 메뉴에 “근태 신청” 추가
2. 일반 직원 로그인 시 “근태 신청”만 활성화
3. 근태 신청에서는 달력에서 날짜를 선택하고
   - 휴가
   - SL
   - 연차
   중 하나를 선택해서 신청
4. 사유는 필수 아닌 선택 입력으로 처리
5. 공휴일은 직원 신청 항목이 아니라 시스템 자동 배정값으로 처리
6. 공휴일은 무조건 쉬는 날이 아니라
   수요 예측과 최소 운영 인원 충족 여부를 먼저 보고 가능할 때만 공휴로 쉬게 반영
7. 시스템이 먼저 14일 동안 4일 휴무 규칙을 충족하는 기본 스케줄을 생성
8. 그 후 직원 신청을 가능한 범위 내 자동 반영
9. 근태 관리는 전건 수동 승인 화면이 아니라
   자동 반영 결과와 예외를 관리하는 화면으로 재구성
10. 근태 관리 담당자는 직원별 남은 휴가 / 연차 수량을 볼 수 있어야 함

이 기준으로
“근태 신청”과 “근태 관리”가 연결된 하나의 운영 흐름처럼 보이도록
UI와 기능 흐름을 설계해주세요.
