현재 LOTTE HOTELS & RESORTS 관리자용 SaaS의 기존 톤앤매너를 유지한 상태로
로그인 / 회원가입 신청 / 계정 및 접근 관리 / 수정 이력 흐름을 아래 기준으로 구체화해주세요.

중요:
- 기존 시스템의 프리미엄 호텔 운영 관리자용 SaaS 톤앤매너 유지
- 딥 네이비 / 웜 화이트 / 절제된 골드 포인트 유지
- 넓은 여백, 얇은 선, 정돈된 그리드, 실무형 UI 유지
- 소비자용 커뮤니티 가입 화면처럼 보이지 않고 호텔 운영 시스템의 계정 관리 흐름처럼 보여야 함
- 이번 수정은 로그인 화면, 호텔 설정 > 계정 및 접근 관리, 호텔 설정 > 수정 이력에 연결되는 흐름이 자연스럽게 보이는 것이 핵심

[1. 로그인 화면 수정]
현재 로그인 화면에 아래 기능을 추가해주세요.

필수 추가 항목:
1) 비밀번호 찾기 버튼
- “비밀번호를 잊으셨나요?” 버튼 클릭 시 팝업 오픈
- 팝업 제목 예시: 비밀번호 찾기
- 입력 항목:
  - 사번
  - 이메일
  - 호텔 선택
- 하단 버튼:
  - 취소
  - 임시 비밀번호 발급 요청

중요:
- 아이디는 이메일이 아니라 “사번” 기준으로 운영
- 사용자는 가입 시 등록한 이메일로 임시 비밀번호를 받는 구조
- 비밀번호 찾기는 단순 안내가 아니라 실제 재설정 요청 흐름처럼 보여야 함

2) 회원가입 신청 버튼
- 로그인 화면에 “회원가입 신청” 버튼 추가
- 버튼 클릭 시 회원가입 신청 팝업 또는 별도 신청 화면 오픈

회원가입 신청 입력 항목:
- 이름
- 사번
- 이메일
- 연락처
- 호텔
- 부서
- 직급
- 비밀번호
- 비밀번호 확인

중요:
- 아이디는 사번으로 사용
- 직급은 자유 입력이 아니라 드롭다운 선택형으로 처리
- 직급 드롭다운 값은 직원 관리 화면에서 사용 중인 직급 체계를 그대로 가져와서 사용
- 즉, 직원 관리의 직급과 회원가입 직급이 동일한 시스템 안에 있는 것처럼 보여야 함

직급 드롭다운 값:
- 엘크루
- 주니어
- L1-A
- L1-C
- L1-D
- L2-A
- L2-C
- L3
- L4

회원가입 신청 완료 후 상태:
- 바로 가입 완료가 아니라 “승인 대기” 상태로 접수
- 신청 완료 메시지 예시:
  - 회원가입 신청이 접수되었습니다
  - 관리자 승인 후 로그인할 수 있습니다

[2. 비밀번호 정책 반영]
회원가입 시 사용자가 본인이 직접 비밀번호를 설정하고 신청하는 구조이므로,
“첫 로그인 시 비밀번호 변경 강제 여부 설정” 기능은 넣지 말아주세요.

즉:
- 회원가입 단계에서 본인이 비밀번호 설정
- 승인 후 그 비밀번호로 로그인 가능
- 별도의 첫 로그인 강제 변경 설정은 불필요

다만,
비밀번호 초기화가 발생한 경우는 아래 규칙을 따릅니다.

[3. 비밀번호 초기화 방식]
비밀번호 초기화는 아래 방식으로 설계해주세요.

방식:
- 임시 비밀번호 발급
- 첫 로그인 시 비밀번호 변경 강제
- 임시 비밀번호는 가입한 이메일로 발급

즉,
관리자가 비밀번호 초기화를 실행하면
직원에게 등록 이메일로 임시 비밀번호가 발송되는 흐름으로 보이게 해주세요.

권장 UX:
1. 관리자 화면에서 “비밀번호 초기화” 버튼 클릭
2. 확인 팝업 오픈
   - 이 직원의 비밀번호를 초기화하시겠습니까?
   - 등록된 이메일로 임시 비밀번호가 발급됩니다
3. 버튼:
   - 취소
   - 초기화
4. 완료 후 상태 메시지:
   - 임시 비밀번호 발급 완료
   - 첫 로그인 시 비밀번호 변경이 필요합니다

[4. 호텔 설정 > 계정 및 접근 관리 구조]
호텔 설정의 “계정 및 접근 관리” 탭은 아래 3개 영역으로 구성해주세요.

A. 회원가입 승인 대기
- 승인 대기 건수 표시
- 신청자 리스트 테이블
- 컬럼 예시:
  - 이름
  - 사번
  - 이메일
  - 호텔
  - 부서
  - 직급
  - 신청일
  - 상태
  - 관리

관리 액션:
- 승인
- 거절
- 상세 보기

중요:
- 회원가입은 승인 대기 상태로 들어오고
- 관리자가 승인 또는 거절할 수 있어야 함

B. 활성 계정 관리
- 승인된 직원 계정 목록
- 컬럼 예시:
  - 이름
  - 사번
  - 이메일
  - 호텔
  - 부서
  - 직급
  - 권한
  - 계정 상태
  - 관리

관리 액션:
- 비밀번호 초기화
- 권한 변경
- 계정 비활성화
- 상세 보기

C. 비밀번호 초기화 영역
- 별도 설명 카드 또는 계정 목록 액션과 연결
- 현재 화면의 설명 문구만 있는 구조가 아니라
  실제 초기화 버튼이 있는 관리 기능처럼 보여야 함

[5. 관리자 권한 체계]
권한 체계는 아래 3단계로 설계해주세요.

1) 일반 직원
- 근태 신청만 사용 가능

2) 운영 관리자
- 권한 부여 / 회수 기능을 제외한 나머지 관리자 기능 사용 가능
- 즉, 아래는 사용 가능
  - 근태 관리
  - 직원 관리
  - 수요 예측 관련 관리
  - 호텔 설정 일부 관리
  - 회원가입 승인
  - 비밀번호 초기화
  - 운영 정책 확인/수정 가능
- 단, 다른 사용자에게 권한을 부여하거나 회수하는 기능은 불가

3) 최종 관리자
- 기존 “슈퍼 관리자” 명칭 대신 “최종 관리자”로 변경
- 모든 기능 사용 가능
- 권한 부여 / 회수 포함 전체 권한 보유

중요:
- 운영 관리자는 대부분의 운영 기능을 사용 가능
- 최종 관리자만 권한 부여/회수 가능
- “슈퍼 관리자”라는 표현은 사용하지 말고 모두 “최종 관리자”로 통일

[6. 권한 부여 / 회수 기능]
권한 부여 / 회수는 활성 계정 관리 영역 또는 상세 패널에서 가능하게 해주세요.

권한 변경 팝업 예시:
- 직원명
- 현재 권한
- 변경 권한 드롭다운
  - 일반 직원
  - 운영 관리자
  - 최종 관리자
- 저장 / 취소 버튼

중요:
- 권한 변경 액션은 최종 관리자에게만 노출되는 것처럼 보여야 함
- 운영 관리자 화면에서는 권한 변경 버튼이 비활성 또는 숨김 처리

[7. 수정 이력 화면 강화]
호텔 설정 > 수정 이력 화면은 단순히 “관리자가 수정함” 수준이 아니라
누가 어떤 권한으로 무엇을 바꿨는지 추적 가능하게 강화해주세요.

반드시 포함할 컬럼:
- 관리자 이름
- 관리자 권한
- 작업 유형
- 대상
- 변경 내용
- 일시

[8. 공통 상태 피드백]
아래 액션들은 모두 실행 후 결과가 분명히 보이게 해주세요.

적용 대상:
- 회원가입 신청
- 회원가입 승인 / 거절
- 비밀번호 초기화
- 권한 변경
- 계정 비활성화

표시 예시:
- 신청 접수 완료
- 승인 완료
- 거절 완료
- 임시 비밀번호 발급 완료
- 권한 변경 완료

즉,
버튼을 눌렀는데 끝나는 구조가 아니라
반영 결과가 사용자에게 즉시 전달되게 해주세요.

[9. 디자인 방향]
반드시 아래 느낌을 유지해주세요.
- 로그인 화면은 프리미엄 호텔 브랜드의 차분한 로그인 화면
- 계정 관리 화면은 관리자 콘솔답게 실무적이고 정돈된 구조
- 승인/초기화/권한 변경 액션은 분명하지만 과하게 공격적인 느낌은 피할 것
- 수정 이력 화면은 감사 로그처럼 신뢰감 있게 보이게

[10. 최종 요청]
현재 시스템을 아래 기준으로 수정해주세요.

- 로그인 화면에 비밀번호 찾기 버튼 추가
- 비밀번호 찾기 팝업에서 사번, 이메일, 호텔 선택 후 임시 비밀번호 발급 요청 가능
- 로그인 화면에 회원가입 신청 버튼 추가
- 회원가입 신청 시 아이디는 사번으로 사용
- 직급은 직원 관리의 직급 체계를 그대로 가져와 드롭다운 선택형으로 구성
- 사용자는 가입 시 본인이 비밀번호를 직접 설정
- 따라서 첫 로그인 시 비밀번호 변경 강제 설정은 회원가입 흐름에 넣지 않음
- 비밀번호 초기화는 임시 비밀번호 발급 + 첫 로그인 시 변경 강제 방식으로 구성
- 임시 비밀번호는 가입한 이메일로 발급
- 운영 관리자 권한은 권한 부여/회수를 제외한 대부분의 관리자 기능 사용 가능
- 슈퍼 관리자 명칭은 모두 “최종 관리자”로 변경
- 최종 관리자만 권한 부여/회수 가능
- 수정 이력에는 누가, 어떤 권한으로, 무엇을 수정했는지 추적 가능하게 강화

이 기준으로 로그인, 계정 관리, 권한 관리, 수정 이력이 하나의 연결된 관리자 시스템처럼 보이도록 UI와 흐름을 설계해주세요.

핵심: 아이디는 사번, 직급은 직원 관리 직급 드롭다운 재사용, 비밀번호 초기화는 이메일 임시 비밀번호 발급, 권한 체계는 일반 직원 / 운영 관리자 / 최종 관리자 3단계로 정리해주세요.
