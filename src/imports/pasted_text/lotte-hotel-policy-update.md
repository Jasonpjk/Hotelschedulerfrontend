현재 LOTTE HOTELS & RESORTS 관리자용 SaaS의 기존 톤앤매너를 유지한 상태로
“호텔 설정”, “근태 신청”, “근태 관리”와 연결되는 운영 정책 및 신청 로직을 아래 기준으로 반영해주세요.

중요:
- 기존 시스템의 프리미엄 호텔 운영 관리자용 SaaS 톤앤매너 유지
- 딥 네이비 / 웜 화이트 / 절제된 골드 포인트 유지
- 넓은 여백, 얇은 선, 정돈된 그리드, 실무형 UI 유지
- 소비자용 일정 앱이 아니라 호텔 운영 시스템처럼 보여야 함
- 이번 수정은 디자인만이 아니라 정책 구조와 신청/반영 흐름이 화면에 명확히 드러나야 함

[1. 호텔 설정 페이지에 반드시 들어갈 핵심 기능]
호텔 설정 페이지는 단순 환경설정이 아니라
스케줄 생성 엔진의 기준과 사용자 접근 권한을 관리하는 관리자 페이지로 설계해주세요.

필수 기능:
1) 직원 회원가입 승인 기능
- 직원이 근태 신청을 위해 회원가입 신청하면 승인/거절할 수 있어야 함
- 승인 대기 / 승인 완료 / 비활성 상태 구분 가능
- 기본 계정 정보 확인 가능

2) 기본 운영 정책 추가 / 변경 / 삭제 기능
- 운영 규칙을 필수와 권장으로 나누어 관리
- 카드 또는 섹션 구조로 보기 쉽게 정리
- 정책별 수정 / 삭제 / 추가 가능

3) 최소 조별 인원 설정
- 오전조 / 오후조 / 야간조 / 중간조 각각 최소 인원 설정 가능
- 필수 운영정책과 직접 연결되게 구성

4) 직원 로그인 비밀번호 초기화 기능
- 관리자만 실행 가능
- 직원 계정 목록에서 초기화 가능
- 보안상 관리자 기능처럼 보이게

[2. 기본 운영정책 구조]
운영정책은 반드시 “필수”와 “권장”으로 구분해주세요.

A. 필수 운영정책
아래 항목은 반드시 반영되는 규칙으로 구성해주세요.

1. 월요일 기준 14일 동안 휴무 4회 필수
- 롤링 14일 단위가 아니라
- 월요일을 시작점으로 한 14일 블록 기준으로 휴무 4회가 반드시 들어가야 함

2. 최소 조별 인원 반드시 충족
- 각 조에 설정된 최소 인원은 스케줄 생성 시 반드시 충족

3. 각 조별 인차지 1명 이상 필수
- 오전조 / 오후조 / 야간조 / 중간조마다 최소 1명 이상의 인차지 배치
- 인차지 중복 자체는 문제 없음

4. 해당 월 야간조 담당자는 월 내 타 조 전환 불가
- 해당 월에 야간조로 운영되는 직원은
  그 월 안에서는 오전조 / 오후조 / 중간조로 전환하지 않음

5. 전월 야간조 담당자가 익월에 타 조로 전환될 경우
   익월 첫 2일은 OFF 강제 배정 후 투입 가능
- 예:
  3월 야간조 직원은 3월 안에서는 야간조 유지
  4월에 오전조/오후조/중간조로 전환 가능
  단, 4월 1일~2일은 OFF를 강제 배정한 후 타 조로 투입 가능

6. 야간조가 아닌 일반 조 전환에는 최소 휴무 규칙 없음
- 오전조 ↔ 오후조 ↔ 중간조 전환에는
  별도 최소 휴무일 강제 규칙 없음

7. SL 연속 사용 불가
- SL은 여성 보건휴가로
- 연속 날짜로 신청할 수 없도록 차단
- 근태 신청 화면에서도 연속 선택 시 즉시 안내 표시

B. 권장 운영정책
아래 항목은 반드시 막는 규칙이 아니라
경고 또는 지양 규칙으로 처리해주세요.

1. 오후조 출근 후 다음날 오전조 투입 지양
- 강제 차단이 아니라 권장 규칙
- 위반 시 경고 또는 주의 표시

2. 최대 연속 근무 6일 초과 시 경고
- 6일 초과를 절대 금지하지는 않음
- 다만 경고 문구가 뜨고 검토 필요 대상으로 보이게

[3. 근태 신청 화면에 추가할 기능]
근태 신청 화면은 일반 직원이 사용하는 화면입니다.
아래 내용을 반드시 반영해주세요.

1) 사유 입력은 선택
- 근태 신청 시 사유는 필수가 아니라 선택 입력
- UI 라벨도 “사유(선택)”으로 명확히 표시

2) 신청 가능한 유형
- 휴가
- SL
- 연차

중요:
- 공휴 / 휴무는 직원이 직접 신청하는 항목이 아님
- 공휴 / 휴무는 자동 스케줄 엔진이 판단하여 반영하는 값

3) 본인 기준 잔여 휴가 / 잔여 연차 입력 기능 추가
- 직원이 본인 계정에서 직접 입력 가능하게
- 근태 신청 시 본인 잔여 휴가와 잔여 연차를 입력
- 최초 입력 후 이후 신청 시 자동 불러오기 가능하게 표현
- 이 값은 “직원 입력 기준”처럼 보이게 정리

표시 예시:
- 휴가 잔여(직원 입력): 15일 중 5일
- 연차 잔여(직원 입력): 15일 중 12일

4) 주말 / 공휴 선호 입력 기능 추가
직원이 근태 신청 시 아래 선호값도 입력할 수 있게 해주세요.

주말 선호:
- 주말 휴무 희망
- 주말 근무 희망
- 상관없음

공휴 선호:
- 공휴 휴무 희망
- 공휴 근무 희망
- 상관없음

중요:
- 이것은 강제 신청이 아니라 선호값
- 자동 스케줄 생성 시 참고하는 보조 조건으로 보이게

[4. 공휴일 처리 원칙]
공휴일은 무조건 쉬는 날이 아닙니다.
이 원칙이 화면과 설명에 명확히 드러나야 합니다.

반드시 아래처럼 반영해주세요.
- 공휴일이라도 호텔 운영이 우선
- 수요 예측과 최소 운영 인원 기준을 먼저 판단
- 최소 인원만 출근해도 되는 상황이면 일부 직원은 공휴로 쉬도록 자동 반영
- 최소 인원 충족이 어려우면 공휴일에도 근무 배치 가능

즉,
공휴일은 절대 우선 보장값이 아니라
운영 조건을 먼저 충족한 후 가능할 때 적용되는 자동 반영값처럼 보여야 합니다.

[5. 공정성 계산 제외 규칙]
주말/공휴 선호값은 공정성 계산에도 영향을 줍니다.

반드시 반영할 규칙:
- 본인이 “주말 근무 희망”을 선택한 경우
  → 주말 휴무 공정성 계산 대상에서 제외

- 본인이 “공휴 근무 희망”을 선택한 경우
  → 공휴일 휴무 공정성 계산 대상에서 제외

- “상관없음” 선택자는 기본 공정성 계산 대상 유지

즉,
공정성 리포트나 근무표 생성 로직에서는
개인 선호를 반영한 뒤 공정성 계산 대상을 자동 조정하는 구조로 설계해주세요.

[6. 자동 스케줄 생성 우선순위]
근태 신청, 근태 관리, 수요 예측, 호텔 설정은 아래 순서로 연결되어야 합니다.

자동 반영 우선순위:
1. 월요일 기준 14일 동안 휴무 4회
2. 최소 조별 인원 충족
3. 각 조별 인차지 1명 이상 배치
4. 해당 월 야간조 직원의 월 내 조 고정
5. 전월 야간조 직원의 익월 첫 2일 OFF 강제 배정
6. 공휴일 운영 가능 여부 판단
7. 직원이 신청한 휴가 / SL / 연차 반영
8. 직원의 주말/공휴 선호 반영
9. 권장 규칙 위반 시 경고 표시

중요:
- 직원 신청이 먼저가 아니라
- 기본 운영과 법칙이 먼저 충족된 후
- 가능한 범위 내에서 신청이 자동 반영되어야 함

[7. 근태 관리 화면에 반드시 드러나야 할 내용]
근태 관리 화면은 승인/반려 중심이 아니라
자동 반영 결과와 예외를 관리하는 화면이어야 합니다.

근태 관리 담당자가 꼭 봐야 하는 정보:
- 직원명
- 신청 날짜 묶음 또는 신청 건수
- 신청 유형
- 최종 반영
- 자동 반영 결과
- 반영 안된 신청 건수
- 반영 안된 직원 수
- 휴가 잔여(직원 입력 기준)
- 연차 잔여(직원 입력 기준)
- 충돌 여부
- 공휴 운영 충족 여부
- 검토 필요 여부

중요:
근태 관리 담당자는
직원별로 남은 휴가와 연차를 볼 수 있어야 하고,
신청이 실제로 무엇으로 반영됐는지도 봐야 합니다.

[8. 호텔 설정 페이지 섹션 제안]
호텔 설정 페이지는 아래처럼 구조화해주세요.

1) 계정 및 접근 관리
- 회원가입 승인
- 계정 상태 관리
- 비밀번호 초기화

2) 필수 운영정책
- 14일 4휴무
- 최소 조별 인원
- 인차지 최소 배치
- 월내 야간조 고정
- 익월 전환 시 2일 OFF
- SL 연속 사용 불가

3) 권장 운영정책
- 오후조 다음날 오전조 지양
- 연속 근무 6일 초과 경고

4) 근태/선호 반영 정책
- 직원 잔여 휴가/연차 입력 방식
- 주말 선호 반영
- 공휴 선호 반영
- 공정성 계산 제외 기준

5) 최근 수정 이력
- 누가 언제 정책을 수정했는지 간단히 표시
- 관리자 콘솔 느낌 유지

[9. 디자인 방향]
반드시 아래 느낌으로 정리해주세요.
- 복잡한 설정 화면이지만 지나치게 기술적이지 않게
- 운영정책 관리 화면처럼 보이게
- 프리미엄 브랜드 관리자 콘솔 느낌
- 카드 남발 금지
- 섹션은 명확하되 과하게 무겁지 않게
- 필수/권장 구분이 시각적으로 분명하게 보이게

[10. 최종 요청]
현재 시스템에 아래 방향이 자연스럽게 반영되도록
호텔 설정, 근태 신청, 근태 관리와 연결된 정책 구조를 설계해주세요.

핵심 반영사항:
- 호텔 설정에 회원가입 승인, 정책 관리, 최소 조별 인원, 비밀번호 초기화 포함
- 운영정책은 필수와 권장으로 구분
- 야간조 규칙은 월 내 고정 / 익월 첫 2일 OFF 후 타 조 전환 가능으로 명확히 분리
- 일반 조 전환에는 최소 휴무 규칙 없음
- 6일 초과는 금지가 아니라 경고
- 직원이 근태 신청 시 본인 잔여 휴가 / 연차 입력 가능
- 사유는 선택 입력
- 주말/공휴에 대해 휴무 희망 / 근무 희망 / 상관없음 입력 가능
- 주말 근무 희망자와 공휴 근무 희망자는 해당 공정성 계산에서 제외
- 공휴일은 운영 우선 판단 후 가능할 때만 쉬도록 자동 반영
- SL은 연속 사용 불가

이 기준으로 정책, 신청, 자동 반영 로직, 관리자 관리 흐름이 서로 연결된 하나의 시스템처럼 보이도록 UI와 구조를 설계해주세요.

중요: 야간조 규칙은 “야간 후 다음 슬롯 금지”가 아니라 “해당 월 내 야간조 고정 + 익월 전환 시 첫 2일 OFF” 규칙으로 정확히 표현해주세요.
