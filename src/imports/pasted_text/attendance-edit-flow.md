현재 LOTTE HOTELS & RESORTS 관리자용 SaaS의 “근태 신청” 화면을 아래 기준으로 수정해주세요.

중요:
- 이번 수정은 “근태 신청” 화면에만 적용해주세요.
- 다른 화면은 건드리지 말아주세요.
- 기존 시스템의 프리미엄 호텔 운영 관리자용 SaaS 톤앤매너 유지
- 딥 네이비 / 웜 화이트 / 절제된 골드 포인트 / 넓은 여백 / 얇은 선 / 정돈된 그리드 유지

[1. 수정 진입 시 기존 신청 날짜 자동 선택]
현재 근태 신청 내역에서 “상세” 버튼을 누르고, 그 안에서 “수정” 버튼을 누르면
수정 화면으로 들어갔을 때 기존에 신청했던 날짜가 자동으로 선택되어 있어야 합니다.

즉,
- 사용자가 다시 날짜를 처음부터 고르지 않아도 되게
- 기존 신청 날짜가 수정 화면의 달력 또는 날짜 선택 UI에 기본값으로 미리 선택된 상태로 보여야 합니다

적용 대상:
- 단일 날짜 신청
- 여러 날짜 일괄 신청

예시:
- 4월 15일 신청 건 수정 진입 → 4월 15일이 자동 선택
- 4월 15일, 4월 16일 일괄 신청 건 수정 진입 → 두 날짜 모두 자동 선택

중요:
- 수정은 신규 작성이 아니라 기존 신청을 편집하는 흐름처럼 보여야 합니다
- 따라서 날짜, 유형, 사유 등 기존 값이 기본값으로 채워져 있어야 합니다

[2. 자동 반영 완료 건 수정 시 기존 데이터 대체]
현재 문제:
자동 반영 완료된 신청 건에서
“상세” → “수정” → 동일 날짜에서 사유 입력 후 “저장”을 누르면
기존 데이터가 유지되고, 수정한 내용이 신규 데이터처럼 추가되고 있습니다.

이 동작은 잘못된 흐름이므로 수정해주세요.

원하는 동작:
- 기존 신청 건을 수정하면
- 신규 신청이 하나 더 추가되는 것이 아니라
- 기존 데이터가 수정된 내용으로 “대체”되어야 합니다

즉,
수정 저장 후 결과는 아래처럼 보여야 합니다.
- 기존 건 1개만 유지
- 그 건의 내용이 수정된 값으로 업데이트
- 중복 행 생성 금지
- 동일 날짜에 기존 건 + 수정 건 2개가 동시에 존재하면 안 됨

[3. 수정 저장 동작을 “업데이트”로 바꿔주세요]
수정 화면에서 “저장” 버튼을 누를 때의 의미를 아래처럼 명확히 바꿔주세요.

현재 잘못된 의미:
- 저장 = 신규 신청 추가

수정 후 올바른 의미:
- 저장 = 기존 신청 업데이트

즉,
수정 저장은 “새 신청 생성”이 아니라
“기존 신청 변경사항 반영”으로 처리되어야 합니다.

[4. 동일 날짜 수정 시 중복 생성 금지]
특히 아래 상황이 매우 중요합니다.

문제 상황:
- 자동 반영 완료된 기존 신청 건 존재
- 상세 → 수정 진입
- 동일 날짜 유지
- 사유만 수정
- 저장 클릭
- 기존 건 유지 + 수정 건 신규 추가됨

수정 목표:
- 동일 날짜에서 사유만 바꾼 경우에도
  기존 건이 수정된 내용으로 교체되어야 함
- 동일 날짜 동일 신청 건의 중복 생성 금지

즉,
수정 후에는 아래처럼 보여야 합니다.
- 수정 전 기존 행 사라짐
- 수정된 내용이 같은 위치의 기존 건을 대체
- 사용자는 “정상적으로 수정되었다”고 이해할 수 있어야 함

[5. 수정 화면 기본값 유지]
상세에서 수정으로 들어갔을 때 아래 값들은 모두 기존 신청 기준으로 기본값이 채워져 있어야 합니다.

필수 기본값:
- 신청 날짜
- 신청 유형(휴가 / SL / 연차)
- 사유
- 선택된 날짜 목록(다중 신청 시)
- 기타 기존 입력값

즉,
수정 화면은 빈 새 신청 폼처럼 보이면 안 되고,
“기존 신청을 편집하는 폼”처럼 보여야 합니다.

[6. 저장 후 사용자 피드백]
수정 저장 후에는 아래 중 하나 이상의 피드백이 보여야 합니다.
- 수정 완료
- 신청 내역이 업데이트되었습니다
- 변경사항 반영 완료

중요:
- 신규 신청이 추가된 것이 아니라
- 기존 신청이 수정되었다는 점이 분명히 느껴져야 합니다

[7. 신청 내역 리스트 반영 방식]
수정 저장 후 “내 신청 내역” 리스트에서도 아래가 지켜져야 합니다.

- 기존 행이 수정된 내용으로 갱신
- 중복 행 추가 금지
- 신청일/상태/사유 등 관련 값이 최신 내용으로 반영
- 자동 반영 완료 상태였던 건도 수정 후 하나의 동일한 신청 건으로 유지

[8. 최종 요청]
현재 “근태 신청” 화면을 아래 기준으로 수정해주세요.

- 근태 신청 내역에서 상세 → 수정 진입 시 기존 신청 날짜가 자동 선택되어 있게 수정
- 다중 날짜 신청도 기존 선택 날짜가 그대로 불러와지게 수정
- 자동 반영 완료된 신청 건을 수정하고 저장할 때 기존 데이터가 유지되면서 신규 추가되지 않게 수정
- 수정 저장은 신규 생성이 아니라 기존 신청 대체/업데이트로 동작하게 변경
- 동일 날짜에서 사유만 수정해도 기존 건이 수정본으로 교체되게 처리
- 수정 후 사용자에게 “업데이트 완료” 피드백이 보이게 정리

핵심은,
“상세 → 수정”은 새 신청 작성이 아니라 기존 신청 편집이어야 하고,
저장 시 기존 데이터가 남아 중복 생성되지 않고 수정본으로 대체되어야 한다는 점입니다.

핵심: 상세 → 수정은 기존 신청 편집 흐름이어야 하며, 저장 시 새 건 추가가 아니라 기존 건 업데이트/대체로 처리해주세요.
